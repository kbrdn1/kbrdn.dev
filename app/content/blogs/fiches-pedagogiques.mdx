## Le contexte

Chez Flippad, on m'a confie un projet ambitieux : la refonte complete de **Fiches Pedagogiques**, une plateforme de ressources educatives destinee aux enseignants du primaire francais, de la Petite Section au CM2. Le site existait deja, mais le code etait devenu difficilement maintenable. Plutot que de patcher indefiniment, la decision a ete prise de tout repartir de zero.

J'etais project lead et responsable du backend. Le projet impliquait de repenser l'architecture, de modeliser un domaine metier riche (fiches, collections, abonnements, paiements, ecoles...) et de construire une API capable de servir un front moderne tout en gerant des volumes de contenu importants -- on parle de milliers de fiches pedagogiques, chacune avec ses fichiers, ses previews, ses metadata SEO, ses relations avec des themes, des domaines scolaires, des competences, etc.

## Pourquoi une refonte from scratch

La base de code d'origine avait accumule des annees de dette technique. Les modeles etaient couples aux vues, la validation etait eparpillee, et chaque nouvelle fonctionnalite demandait de toucher a une dizaine de fichiers sans pattern clair. L'ajout d'un systeme d'abonnement Stripe sur cette base aurait ete du suicide technique.

On avait aussi des besoins nouveaux qui ne rentraient pas dans l'ancien moule :

- Une recherche full-text performante avec filtres combines (niveau scolaire, theme, type de fiche, domaine...)
- Un back-office admin complet avec gestion fine des permissions
- Un systeme de paiement flexible (abonnements + achats unitaires + licences ecoles)
- Du media management avance (previews auto, conversion WebP, extraction de miniatures PDF/video)

Repartir de zero nous permettait de poser des fondations solides et d'aller vite sur le long terme.

## Le stack technique

### Backend -- Laravel 12, PHP 8.4

Le choix de Laravel s'est impose naturellement. L'ecosysteme est mature, la documentation est excellente, et surtout il offre tout ce qu'on avait besoin out of the box : Sanctum pour l'auth, Cashier pour Stripe, des jobs en queue, un ORM puissant.

On tourne sur **PHP 8.4** pour profiter des enums natifs, des readonly properties, et des performances ameliorees. La base de donnees est **MySQL** avec **Redis** reparti sur 5 bases logiques :

- DB 0 : sessions et donnees par defaut
- DB 1 : cache applicative
- DB 2 : cache de recherche
- DB 3 : logs structures
- DB 4 : media processing

**Elasticsearch 8.16** gere toute la recherche full-text avec 38 index dedies et du fuzzy matching.

### Frontend -- Nuxt 4, Vue 3, TypeScript

Le front est une application Nuxt 4 avec Vue 3 et TypeScript. L'UI est construite avec **shadcn-nuxt** et **TailwindCSS 4**, la validation des formulaires passe par **Zod + VeeValidate**, et l'etat global est gere avec **Pinia**. L'authentification est assuree par **nuxt-auth-sanctum** qui dialogue avec l'API Sanctum cote backend.

## L'architecture modulaire

C'est probablement la decision technique dont je suis le plus satisfait. Plutot qu'un monolithe Laravel classique avec des controllers geants, j'ai mis en place une **architecture modulaire** ou chaque entite metier est un module autonome.

Chaque module admin contient :

- Un **Controller** qui herite du CRUDController generique
- Un **Repository** pour les requetes
- Une **Resource** pour la serialisation
- Un dossier **Strategies** avec trois classes : `RuleStrategy` (validation), `RelationStrategy` (relations), `ExceptionStrategy` (gestion d'erreurs)
- Un dossier **Exceptions** propre au module

```
app/Http/Modules/Admin/Items/
├── ItemController.php
├── ItemRepository.php
├── ItemResource.php
├── Strategies/
│   ├── ItemRuleStrategy.php
│   ├── ItemRelationStrategy.php
│   └── ItemExceptionStrategy.php
└── Exceptions/
```

Cette structure se repete pour chacun des **58 modules admin**. Ca peut sembler verbose, mais en pratique c'est un gain enorme : quand je dois intervenir sur les fiches pedagogiques, tout est dans `Modules/Admin/Items/`. Pas besoin de naviguer entre `Controllers/`, `Requests/`, `Resources/` et `Policies/` a la racine.

### Le CRUDController generique

Le coeur du systeme est un `CRUDController` abstrait qui fournit toutes les operations CRUD standard :

```php
class CRUDController extends Controller
{
  protected ModulesEnum $name;
  protected PaginationService $pagination;
  protected RuleContext $rules;
  protected Repository $repository;
  protected Resource $resource;

  public function __construct(
    ModulesEnum $name,
    Repository $repository,
    Resource $resource,
  ) {
    $this->name = $name;
    $this->rules = new RuleContext($name);
    $this->repository = $repository;
    $this->resource = $resource;
    $this->pagination = Pagination::getConfiguration($name)
      ?? new PaginationService();
  }
}
```

Les controllers concrets n'ont plus qu'a declarer leurs specificites. Par exemple, `ItemController` ajoute juste la sanitization HTML et la conversion de certains champs :

```php
class ItemController extends CRUDController
{
  use SanitizesHtmlContent;

  protected array $htmlFields = ["description", "meta_description"];
  protected array $strictHtmlFields = ["meta_description"];

  public function __construct(
    ItemRepository $repository,
    ItemResource $resource,
  ) {
    parent::__construct(ModulesEnum::ITEM, $repository, $resource);
  }

  public function store(Request $request): JsonResponse
  {
    $this->convertStringToArray($request);
    $this->sanitizeHtmlContent($request);
    return parent::store($request);
  }
}
```

### Le pattern Strategy pour la validation

La validation est un point critique dans une app avec 112 tables et des dizaines de formulaires. Plutot que des `FormRequest` classiques, j'ai implemente un **pattern Strategy** ou chaque module definit ses regles via une `RuleStrategy`.

Chaque strategy expose trois methodes : `getPrimitiveRules()` pour le filtrage et les updates partiels, `getRules()` pour la creation avec toutes les contraintes, et `getDataCollectionRules()` pour les imports en masse.

```php
class ItemRuleStrategy extends RuleStrategy
{
  public function getRules(): array
  {
    return [
      "title" => "required|string|max:255",
      "cost" => "required|integer",
      "slug" => ["nullable", "string", "regex:/^[a-z0-9-]+$/",
                  "unique:items,slug"],
      "description" => ["string", "nullable", new SanitizedHtml()],
      "school_domains" => "required|array",
      "school_domains.*" => "integer|exists:school_domains,id",
      "sub_school_domains.*" => [
        "integer",
        "exists:sub_school_domains,id",
        $this->getSubSchoolDomainRule(
          request()->input("school_domains")
        ),
      ],
      "collections" => [
        "array",
        new CollectionRequiresAllowedItemType(),
      ],
      // ...
    ];
  }
}
```

Les regles custom comme `SubSchoolDomainBelongsToSchoolDomain` ou `CycleBelongsToSection` garantissent la coherence des donnees metier directement au niveau de la validation -- un sous-domaine scolaire doit appartenir au domaine parent selectionne, un cycle doit correspondre a la section choisie.

## Le DepthAwareResource -- resoudre le N+1 autrement

Un des problemes classiques des API REST avec des relations imbriquees, c'est l'explosion des requetes (N+1) ou les references circulaires. Si une fiche a des collections, et qu'une collection reference ses fiches, on peut vite boucler indefiniment.

J'ai concu un trait `DepthAwareResource` qui resout ce probleme avec un systeme de profondeur et de detection de cycles :

```php
trait DepthAwareResource
{
  protected const MAX_ABSOLUTE_DEPTH = 3;

  protected function createDepthAwareRelations(
    array $relationMap,
    $item,
  ): array {
    if ($this->currentDepth >= self::MAX_ABSOLUTE_DEPTH) {
      return [];
    }

    if (!$this->shouldIncludeRelationsAtCurrentDepth()) {
      return [];
    }

    $this->registerSelfInAncestorChain();

    $relations = [];
    foreach ($relationMap as $relationKey => $module) {
      if ($this->wouldCreateCircularReference($module)) {
        continue;
      }

      $ancestorChainSnapshot = $this->getModuleChain();
      $this->registerModuleInChain($module);

      $context = $this->createRelationContext($module);
      $relations[$relationKey] = $this->retrieveRelation(
        $relationKey,
        static fn() => $context->getFromModel($item),
      );

      $this->setModuleChain($ancestorChainSnapshot);
    }

    return $relations;
  }
}
```

Le mecanisme utilise une chaine d'ancetres stockee dans les attributs de la requete Laravel. Quand une resource serialise ses relations, elle enregistre son module dans la chaine. Si un module enfant detecte qu'il remonte vers un ancetre deja present, il coupe la branche. Et grace a l'isolation par branche (le snapshot/restore), deux relations soeurs ne s'influencent pas mutuellement.

Le front peut aussi controler la profondeur via un parametre `depth` et selectionner les champs retournes via un header `fields` -- utile pour les listes ou on n'a besoin que de `id`, `title` et `thumbnail`.

## La recherche full-text avec Elasticsearch

La recherche est au coeur de l'experience utilisateur. Un enseignant de CE2 qui cherche "fractions decimales" doit trouver des resultats pertinents en quelques millisecondes, avec la possibilite de filtrer par niveau, theme, type de fiche, etc.

Le service de recherche est construit en couches :

- Un **SearchCriteriaTransformer** qui convertit les parametres de requete en filtres Elasticsearch
- Un **FilterManager** qui applique les filtres combines
- Un **SearchHighlighter** pour surligner les termes dans les resultats
- Un **SearchCacheService** avec cache Redis et invalidation par tags
- Un **ABTestingService** pour tester differents algorithmes de scoring

```php
class SearchService
{
  public function search(
    SearchCriteria $criteria,
    string $context = "search",
  ): array {
    $startTime = microtime(true);

    $filters = $this->transformer->transform($criteria);

    if ($this->needsGlobalSorting($filters)) {
      return $this->searchWithGlobalSorting(
        $criteria, $filters, $context,
      );
    }

    return SearchCache::remember(
      $criteria->toArray(),
      fn() => $this->performSearch($filters, $context),
      "search",
    );
  }
}
```

Chaque modele a son propre index Elasticsearch via le package ElasticLens. Les 53 modeles sont indexes, ce qui permet une recherche cross-entity rapide.

## Le media pipeline

Le projet gere enormement de fichiers : des PDFs de fiches pedagogiques, des images de preview, des videos. Le pipeline media est construit sur **Spatie MediaLibrary** avec des personnalisations maison :

- **CustomPathGenerator** et **CustomUrlGenerator** pour le stockage S3
- **MediaConversionService** qui genere automatiquement des previews en JPG et WebP
- **HeroBannerConversionService** pour les bannieres avec des formats responsifs
- **PdfPageCountService** pour extraire le nombre de pages des PDFs
- Integration **FFmpeg** pour les previews video

Le tout est traite en queue pour ne pas bloquer les requetes. Quand un admin uploade une fiche avec ses fichiers, un job se charge de generer toutes les conversions en arriere-plan.

## Le front -- une base CRUD generique en TypeScript

Cote front, on a applique la meme philosophie DRY qu'au back. La classe `BaseCRUD<TEntity, TPayload, TFilter>` fournit toutes les operations standard a chaque store Pinia :

```typescript
export default class BaseCRUD<
  TEntity extends BaseEntity,
  TPayload extends object,
  TFilter extends object,
  TFields extends readonly (keyof WithCount<TEntity>)[],
> {
  protected readonly basePath: string
  protected readonly fields: TFields
  protected readonly optionFields: readonly (keyof WithCount<TEntity>)[]
  protected readonly optionsLimit: number
  protected transformFn?: <T extends TEntity>(item: T) => T
  // ...
}
```

Les generiques TypeScript garantissent le typage de bout en bout. Quand on cree un `itemStore`, le compilateur sait exactement quels champs sont disponibles, quel est le format du payload de creation, et quels filtres sont applicables. Ca elimine toute une categorie de bugs a la compilation.

La validation cross-layer est un point sur lequel on a beaucoup investi. Les schemas Zod cote front doivent mirrorer les `RuleStrategy` cote back. On a meme un agent QA dedie (`fp-qa`) dans notre workflow Claude Code qui valide la coherence entre les trois couches : Zod, Laravel Rules, et contraintes de la base de donnees.

## Les paiements et les ecoles

Le systeme de monetisation est plus complexe qu'un simple paywall. On gere :

- Des **abonnements individuels** via Stripe (Laravel Cashier)
- Des **achats unitaires** de fiches
- Des **licences ecoles** avec mandats administratifs, invitations de collaborateurs, et gestion de domaines scolaires
- Des **codes promotionnels** avec regles d'utilisation
- Des **comptes demo** a duree limitee

Chaque type de paiement a son propre flow, ses propres webhooks Stripe, et ses propres regles de validation. Les montants sont stockes en centimes dans toute l'API (4450 = 44.50 EUR) pour eviter les problemes d'arrondi en virgule flottante.

## Le routing dynamique

Le routing de l'API est entierement dynamique. Les routes admin sont generees a partir de l'enum `ModulesEnum` :

```php
Route::prefix("admin")
  ->middleware(["auth:sanctum", "crud", "role:super_admin|admin|comptable"])
  ->group(function () {
    $adminRoutes = ModulesEnum::getModules(true);

    foreach ($adminRoutes as $route) {
      if (!file_exists(base_path("routes/admin/{$route}.php"))) {
        continue;
      }
      include base_path("routes/admin/{$route}.php");
    }
  });
```

Ajouter un nouveau module, c'est creer le dossier dans `Modules/Admin/`, ajouter l'entree dans l'enum, et creer le fichier de routes. Le middleware `crud` gere automatiquement les permissions en fonction du module et du role de l'utilisateur.

## Ce que j'en retiens

Ce projet m'a appris plusieurs choses sur l'architecture logicielle a l'echelle :

**Le pattern modulaire paie sur la duree.** Au debut, creer un module complet pour chaque entite semble lourd. Mais des qu'on depasse 15-20 entites, le gain en lisibilite et en maintenabilite est indeniable. Chaque developpeur sait exactement ou trouver le code qui l'interesse.

**Investir dans les abstractions generiques tot.** Le `CRUDController`, le `BaseCRUD` front, le `DepthAwareResource` -- ces composants ont demande du temps a concevoir, mais ils ont accelere la livraison de chaque fonctionnalite suivante. Quand un nouveau module admin prend 15 minutes a creer plutot que 2 heures, l'investissement initial est largement amorti.

**La validation cross-layer est un vrai sujet.** Maintenir la coherence entre la validation front (Zod), la validation back (Laravel Rules), et les contraintes de base de donnees est un defi permanent. L'automatiser autant que possible -- meme partiellement -- evite des bugs subtils en production.

**Elasticsearch n'est pas gratuit.** Indexer 53 modeles et maintenir 38 index demande un investissement significatif en configuration, monitoring et debug. Mais pour une plateforme de contenu ou la recherche est centrale, c'est un investissement qui en vaut la peine.

Ce projet represente plusieurs mois de travail et reste un des plus complets que j'ai menes en tant que lead technique. De la modelisation initiale du domaine metier au deploiement en production, chaque couche a ete pensee pour etre maintenable, extensible, et performante.
