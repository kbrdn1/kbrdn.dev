## Le problème avec les git worktrees

J'utilise les git worktrees presque tous les jours. C'est la meilleure façon que je connaisse de travailler sur plusieurs branches en parallèle sans payer le coût d'un `git stash`, d'un `git checkout`, ou d'un `cargo build` complet à chaque bascule. Une branche = un dossier physique = un workspace isolé avec sa propre `target/`, ses propres dépendances installées, sa propre config locale. Tout cohabite, rien ne s'écrase.

Mais le workflow _autour_ des worktrees est aussi pénible que la primitive est puissante.

Sur un projet Laravel, on avait un script bash dans `tools/worktree-manager.sh` qui orchestrait la création : `git worktree add`, puis copie du `.env.testing`, puis vérification que le `.env` copié ne pointait pas vers AWS RDS (incident vécu — `php artisan test` avec `RefreshDatabase` qui tourne sur la base de prod, on s'est fait prendre une fois, je n'ai pas envie de revivre ça), puis suppression du symlink `vendor/` qui casse l'autoload quand il est partagé entre worktrees, puis `composer install`, puis `direnv allow`.

Le script faisait le job, mais il était figé dans ce projet, avec sa logique anti-RDS hardcodée. Quand j'ai voulu appliquer la même hygiène à `kbrdn.dev` (Nuxt + Bun) ou à gwm lui-même (Rust), j'ai dû tout dupliquer en l'adaptant. Le wrapper Node aurait dû avoir sa logique anti-`node_modules`-symlinké, le wrapper Rust son partage incrémental de `target/`. Trois scripts, trois philosophies, trois sources de bugs — et à chaque modif, j'oubliais de propager.

Je cherchais un outil qui :

- pose la même convention de branche (`<type>/#<issue>-<desc>`) dans **tous** mes projets,
- exécute un bootstrap **déclaratif** lisible par n'importe qui qui ouvre le repo,
- ouvre une **TUI** quand je veux naviguer entre worktrees sans taper trois `git worktree list | grep`,
- soit **un seul binaire**, sans dépendance Python/Node/Bash en chaîne, qui se compile partout.

J'ai écrit **gwm**. La version 0.9.0 est sortie et c'est elle que je décris ici.

## C'est quoi gwm

![gwm — la TUI ratatui avec la sidebar de détails](/images/blog/gwm/banner.png)

`gwm` (pour _git worktree manager_) est un outil CLI + TUI en Rust qui pilote `git worktree` via `libgit2` vendored. Un binaire unique, zéro dépendance à l'exécution — pas même le CLI `git` ou `gwq`. Le tour en 30 secondes :

```bash
cd /chemin/vers/ton/repo
gwm init                                   # écrit un .gwm.toml par défaut
gwm create feat 42 user-authentication     # → ~/cc-worktree/<repo>/feat-42-user-authentication
                                           # → branche feat/#42-user-authentication
                                           # → bootstrap exécuté selon .gwm.toml
gwm                                         # ouvre la TUI sur le repo courant
gcd auth                                    # fuzzy-jump dans le worktree (via `gwm shell-init`)
```

Sans argument, `gwm` ouvre la TUI ratatui : une table avec le nom du worktree, sa branche, son status (clean / dirty / `↑N ↓M` vs upstream) et son chemin, plus une sidebar de détails (commits récents, fichiers modifiés, raccourcis). Les sous-commandes servent les scripts et la CI.

Tout vit dans un crate Rust unique avec une séparation `bin` / `lib` stricte : tous les tests d'intégration vivent sous `tests/` (58 fichiers, **1277 tests** à la v0.9.0), jamais en `#[cfg(test)] mod tests` inline. Ils créent un vrai repo git dans un `TempDir`, exécutent l'opération, et vérifient l'état on-disk + libgit2.

## Pourquoi Rust et pourquoi libgit2 vendored

Rust règle plusieurs problèmes d'un coup.

**Un binaire compilé unique**, qui se distribue par `cargo install` ou par téléchargement d'archive. Pas de runtime à provisionner. Le pipeline de release shippe cinq cibles standard (Linux x86_64 + aarch64, macOS Intel + Apple Silicon, Windows x86_64) en un workflow GitHub Actions.

**libgit2 vendored** est le détail qui change tout. La crate `git2` peut linker contre une libgit2 système ou la compiler depuis ses sources. La feature `vendored-libgit2` choisit la deuxième : `cargo install gwm` compile libgit2 depuis ses sources C et le lie statiquement dans le binaire. Aucune dépendance partagée à `libgit2.so` côté utilisateur, aucun risque de version incompatible côté CI. Plus lent à compiler la première fois, mais une fois le binaire en place il marche partout, isolé du système.

```toml
[dependencies]
git2 = { version = "0.21", default-features = false, features = ["vendored-libgit2"] }
```

Le reste du stack est plus convenu : `clap` 4.5 (derive) pour le CLI, `ratatui` 0.30 + `crossterm` 0.29 pour la TUI, `serde` + `toml` pour `.gwm.toml`, `nucleo-matcher` pour le fuzzy, `sha2` pour le trust ledger (j'y reviens), `thiserror` pour l'enum d'erreur unifiée. MSRV : Rust 1.82.

## La feature signature : le bootstrap déclaratif

Le cœur de gwm, c'est le `.gwm.toml` au root du repo. Au lieu d'un script bash qui sait tout, on _déclare_ ce que doit faire le bootstrap après chaque création de worktree. Et depuis la 0.9.0, la même config peut vivre en global dans `~/.config/gwm/config.toml`, deep-mergée _sous_ le fichier du repo (le repo gagne toujours sur conflit) — on règle une préférence une fois, tous les repos en héritent.

```toml
[worktree]
base           = "{home}/cc-worktree/{repo}"
path_pattern   = "{type}-{issue}-{desc}"
branch_pattern = "{type}/#{issue}-{desc}"

# 1. Copies depuis le repo principal vers le worktree neuf
[[bootstrap.copy]]
from = ".env"
to   = ".env"
required = false
guards = ["no-aws-rds"]

# 2. Garde-fous regex sur les fichiers copiés
[[bootstrap.guard]]
name = "no-aws-rds"
deny_patterns = ["amazonaws\\.com", "\\.rds\\."]
on_match = "seed-from-example"     # ou "abort"
example_file = ".env.example"

# 3. Symlinks refusés (chaque worktree garde son dossier local)
[[bootstrap.no_symlink]]
path = "target"     # cargo : un target/ partagé fait recompiler tout le monde

# 4. Hooks de cycle de vie, avec prédicats composables
[[hooks.post_create]]
name = "install (bun si dispo, sinon npm)"
run  = "bun install"
when = "file_exists:package.json && cmd_exists:bun"
on_fail = "warn"
```

Chaque primitive répond à un problème concret. Les **copies** déplacent les `.env*` et autres fichiers non-versionnés du repo principal vers le worktree. Les **guards** grep le fichier avant de le copier : si un pattern interdit matche (un hostname AWS RDS qui sent la prod), gwm _refuse_ la copie et seed depuis `.env.example`, ou abort, selon `on_match`. Les **no_symlink** suppriment les symlinks hérités qui ne devraient jamais être partagés entre worktrees — `target/` pour Rust, `vendor/` pour PHP, `node_modules/` pour Node. Les **hooks** sont les phases du cycle de vie : `pre_create`, `post_create`, `pre_bootstrap`, `post_bootstrap`, `pre_remove`, `post_remove`, chacune avec un `on_fail = abort | warn | ignore`.

La grosse évolution depuis les débuts, c'est le moteur de prédicats `when`. Au lieu du seul `file_exists:` d'origine, on a maintenant cinq atomes — `file_exists`, `cmd_exists`, `env_set`, `env_eq`, `glob_exists` — composables avec `!`, `&&`, `||` (précédence `! > && > ||`). On peut écrire `file_exists:package.json && !cmd_exists:bun` pour le fallback npm, ou `glob_exists:src/**/*.rs && !env_set:CI` pour ne lancer un build local que hors CI.

Le résultat est lisible par n'importe qui qui ouvre le repo. Pas besoin de connaître Bash, pas besoin de relire 200 lignes de script. La config raconte l'histoire : voilà ce qui se passe quand on crée un worktree ici. Côté TUI, le rapport de bootstrap s'affiche dans un overlay avec un sigil par step :

```
✓ copy .env.testing → .env.testing
· copy .env → .env            (skipped : source absente)
! no-symlink target           (symlink hérité supprimé)
✓ hook: cargo fetch           (1.4s)
· hook: npm install           (skipped : pas de package.json)
```

Le `·` est volontaire : il dit « j'ai considéré ce step, il n'y avait rien à faire ». Important pour un outil qu'on relit un mois plus tard.

## Le garde-fou qui m'a coûté le plus de réflexion : le trust ledger TOFU

Le bootstrap exécute des commandes shell déclarées dans `.gwm.toml`. Or ce fichier est versionné dans le repo. Donc cloner le repo de quelqu'un d'autre et faire `gwm create` dedans revient à exécuter du code arbitraire écrit par l'auteur du repo. Inacceptable en l'état.

La réponse est un **trust ledger TOFU** (_trust on first use_). Au premier `gwm create` / `gwm bootstrap` contre un `.gwm.toml`, gwm affiche la surface de bootstrap (copies, guards, no-symlinks, commandes) et demande confirmation avant de lancer quoi que ce soit. À l'accord, il enregistre la paire `(URL d'origine, sha256 du .gwm.toml)` dans `~/.config/gwm/trust.toml`. Tant que le fichier ne change pas d'un octet, plus de prompt. Au moindre changement de byte, re-prompt.

Le hash porte sur les **octets bruts**, pas sur le TOML parsé — et c'est volontaire :

```rust
// `rm -rf /tmp/` et `rm -rf /tmp /` ne diffèrent que d'un octet
// et se comportent de façon catastrophiquement différente.
// On hashe les bytes, pas le TOML parsé.
```

En CI, où le prompt interactif n'a pas de sens, on bypasse avec `--allow-bootstrap` ou `GWM_ALLOW_BOOTSTRAP=1`. Le mode forensic `--deny-bootstrap` refuse même un fichier déjà approuvé. Et `gwm trust list / revoke / show` permet d'inspecter et révoquer le ledger. C'est le genre de fonctionnalité invisible quand elle marche, mais qui fait la différence entre un outil personnel et un outil qu'on peut recommander.

## L'interface TUI

La TUI affiche une table (NAME / BRANCH / STATUS / PATH) aux largeurs adaptatives, avec une sidebar de détails responsive — affichée si le terminal est assez large, sinon empilée sous la liste (le défaut depuis la 0.9.0), toggle manuel avec `v`. Quand la sidebar a le focus (`Tab`), `j` / `k` scrollent le panneau au lieu de naviguer dans la liste.

Le contenu de la sidebar (commits récents, status) est **caché** sur l'objet `App` et invalidé seulement quand la sélection change ou au refresh. Sans cache, chaque keypress déclencherait deux `git log` et un `git status` sur le worktree sélectionné — la responsivité s'effondre dès qu'on tient `j` enfoncé. Une review l'avait attrapé à l'époque : « the PR says cached, but no caching is implemented ». Honteux et utile.

Trois choses font le quotidien :

- **Les launchers configurables.** `l` suspend gwm et lance le git TUI (lazygit par défaut) fullscreen sur le worktree sélectionné ; on quitte, gwm reprend exactement où il était. `R` lance un outil de review sur la branche contre sa base (gwm-cli utilise `lumen`, mais `claude`, `codex`, `aider` ou `gh` sont aussi câblables via `[review]`).
- **Les thèmes.** Quatre presets role-based — `catppuccin`, `gruvbox`, `tokyo-night`, `claude-dark` — déclinés sur des rôles sémantiques (`branch`, `dirty`, `clean`, `staged`, `modified`, `accent`…) threadés à travers chaque site de rendu. On surcharge n'importe quel rôle en hex via `[theme.overrides]`.
- **Le keymap remappable.** Toute action est rebindable sous `[tui.keys]`, avec support des chords multi-touches (`g g` pour aller en haut), une palette de commandes fuzzy (`:`) qui retrouve chaque action par son nom, et un filtre (`/`) sur la liste.

## La fondation async de la 0.9.0

C'est la vraie nouveauté de cette version. Avant, les opérations longues (refresh de la liste, fetch GitHub, bootstrap, sync) tournaient sur le thread de l'event loop — un bootstrap lent figeait toute la TUI.

La 0.9.0 introduit une **spine async générique** : un pattern unique pour déporter n'importe quelle opération hors du thread d'affichage. Le worker calcule des données `Send` possédées hors-thread, puis poste le résultat sur un canal `mpsc` tagué par une _génération_. Deux propriétés importantes :

- **coalescing** — si une opération du même type est déjà en vol, un second déclenchement ne lance pas un deuxième worker ;
- **late-result drop** — si on relance avant l'arrivée du premier résultat, la génération est incrémentée et le résultat tardif est jeté (fini les données périmées qui écrasent l'état courant).

Concrètement, le refresh (`f`), le bootstrap (`b`), le sync (`S`) et le fetch GitHub passent désormais par cette spine : le spinner de la statusbar s'anime avec un label (`bootstrapping…`), `q` / `Esc` restent réactifs, et la vue Report s'ouvre à la fin. Le trust gate TOFU, lui, reste synchrone sur le thread principal _avant_ que le worker soit lancé — on ne déporte pas une décision de sécurité.

Par-dessus cette spine, trois panneaux in-TUI sont arrivés : un overlay **Command Logs** (`3`) qui montre le transcript des commandes externes, un panneau **Configuration** (`4`) qui affiche le `.gwm.toml` résolu avec l'origine de chaque valeur, et le focus direct des panneaux (`1` / `2`). Plus une touche **open-docs** (`.`) qui ouvre la doc du repo dans le navigateur.

## La surface CLI

La TUI, c'est le confort ; le CLI, c'est ce qui se scripte. gwm expose une trentaine de sous-commandes, dont les principales :

- **Cycle worktree** — `init`, `create <type> <issue> <desc>`, `new <type> <desc>` (sans numéro d'issue), `list`, `remove <pattern> [--delete-branch] [--dry-run]`, `path <pattern>` (pour `cd "$(gwm path foo)"`), `prune`, `bootstrap`, `sync [--merge]`.
- **GitHub** — `link` / `unlink` / `open` (issue ou PR), `pr` (rend le corps de PR depuis un template), `labels push`, `milestones push`. Les branches en `<type>/#<N>-<slug>` s'auto-lient à leur issue, avec détection éphémère de la PR, et le status remonte dans la sidebar via `gh`.
- **Multiplexers & shell** — `switch`, `tmux <pattern> [--split]`, `zellij <pattern> [--split]`, `shell-init <shell>` (qui installe le `gcd` fuzzy-jump et les complétions).
- **Maintenance & filets** — `doctor` (health-check : dry-run du bootstrap, validation de config), `undo` + `history` pour récupérer une suppression ratée, `trust`, `aliases`, `config get/set/list/validate`, `types [--gitmoji]`, `theme`, `completions`.

Les filets de sécurité du quotidien : `--dry-run` sur `remove` / `prune` pour prévisualiser, le countdown de confirmation sur la suppression armée d'une branche, et `gwm undo` pour restaurer un worktree supprimé par erreur.

## Workflow type d'une journée

```bash
# Trois choses en parallèle, chacune dans son worktree isolé
gwm create feat 10 tui-async-spine     # bootstrap auto (cargo fetch + direnv si .envrc)
cd "$(gwm path async-spine)"
# … coder, committer ; `cargo test` n'interfère pas avec les autres builds

gwm                                     # TUI : naviguer, `l` pour lazygit, `S` pour sync
gwm sync async-spine                    # fetch + rebase sur upstream, conflict-safe

gwm remove async-spine --delete-branch  # supprime dossier + admin .git/worktrees/ + branche
```

La convention `<type>/#<issue>-<desc>` est posée automatiquement à partir des trois arguments de `gwm create`. Les types par défaut (`feat`, `fix`, `hotfix`, `docs`, `test`, `refactor`, `chore`, `perf`, `ci`, `build`) sont surchargeables par repo via `[[branch_types]]`, et `gwm types` liste le set résolu en indiquant sa source.

## Industrialiser : tests et release

Construire l'outil, c'est un truc ; l'expédier proprement, c'en est un autre. Le projet repose sur **1277 tests** répartis sur 58 fichiers d'intégration — config (layering repo + global), aliases, hooks, naming, bootstrap, doctor, linking GitHub, launchers, multiplexers, slices d'état de la TUI (keymap, palette, theme, sidebar), journal undo/history, et opérations worktree libgit2 bout-en-bout. La séparation `lib` / `bin` existe précisément pour que ces tests importent l'API publique sans hack. Trois benches `criterion` surveillent le cache de la sidebar et le commit graph.

Côté livraison : une branche `dev` d'intégration, une `pre-release.yml` sur les tags `-rc.N` / `-alpha.N` / `-beta.N`, une `release.yml` sur les tags stables, les cinq cibles construites à chaque fois, et dependabot configuré sur `dev`. La v0.9.0 a été cuite en `rc.1` → `rc.2` → `rc.3` puis promue stable le 7 juin 2026, le code stable étant identique au dernier rc — seul le label change.

## Installer gwm

| Canal | Commande |
|:--|:--|
| Cargo (source) | `cargo install --path .` |
| cargo-binstall | `cargo binstall gwm` _(binaire précompilé, pas de toolchain Rust)_ |
| Homebrew (macOS) | `brew tap kbrdn1/tap && brew install gwm` |
| Nix flake | `nix profile install github:kbrdn1/gwm-cli` |
| Précompilé | [releases GitHub](https://github.com/kbrdn1/gwm-cli/releases) (Linux / macOS / Windows) |

Le code source est sur [github.com/kbrdn1/gwm-cli](https://github.com/kbrdn1/gwm-cli), sous licence MIT. Une fois en place, fais `gwm init` puis `gwm` dans n'importe quel repo git — et vois si ça fait disparaître la friction qu'il a été conçu pour éliminer. C'est le seul critère qui compte pour un outil personnel.
